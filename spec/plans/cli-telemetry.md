# Plan: CLI Telemetry (Parent — Shared Plumbing)

**Status:** Approved
**Source Feature:** cli/telemetry
**Date:** 2026-05-21
**Owner:** alexandertrakhimenok
**Supersedes:** —

## Summary

Decomposes the parent `cli/telemetry` Feature into 8 ordered tasks that build the shared telemetry plumbing — package boundary, install_id lifecycle, persistent state, opt-out evaluation, channel registry + transmission-timeout wrapper, root-command wiring with first-run notice, the `specscore telemetry` subcommand surface, and the `docs/telemetry.md` skeleton. All 12 ACs from the source Feature are covered; no deferrals.

## Approach

Tasks follow a strict bottom-up dependency order: types and boundary first (Task 1), then leaf utilities (install_id, state file), then the evaluator that consumes them (opt-out), then the cross-cutting wrappers (registry, transmission timeout), then the root-command wiring that orchestrates everything, then the user-facing subcommand surface, then the documentation. Each task is sized to one focused work session; Task 1 carries three ACs because they are the same Go package being created in one pass (types + boundary check + constants).

Decomposition principle: every task touches exactly one logical concern. Tasks that need to verify ACs requiring real channels (`transmission-hard-timeout-bounded`, `telemetry-subcommand-status`'s "lists both registered channels" clause) use *fake channels registered in test code* — that's faithful to what the parent owns (the wrapper / dispatcher), and the children's Plans will integration-verify with real channels.

## Tasks

### Task 1: Bootstrap internal/telemetry package — typed event API + boundary check

**Verifies:** cli/telemetry#ac:internal-telemetry-package-location-fixed, cli/telemetry#ac:vendor-sdk-import-confinement-enforced, cli/telemetry#ac:fixed-event-property-keys-enforced-at-compile-time

Create `internal/telemetry/` as a Go package importable at `github.com/specscore/specscore-cli/internal/telemetry`. Declare the typed property struct (closed-enum keys: `command`, `success`, `duration_ms`, `exit_code`, `cli_version`, `os`, `arch`, `caller`, `install_id`, `is_first_run`). Add `internal/telemetry/boundary_test.go` that walks the repo AST and fails when forbidden vendor-SDK imports appear outside `internal/telemetry/`. Verify the package compiles, the boundary test rejects a synthetic violation, and that an attempt to call the public emit API with a `map[string]any` payload fails at compile time.

### Task 2: Install ID lifecycle

**Verifies:** cli/telemetry#ac:install-id-lifecycle

Implement `internal/telemetry/installid.go`: file at `~/.specscore/install_id` (mode `0600` inside dir mode `0700`), UUID v4 generated on first invocation via atomic write-then-rename, per-machine not per-project, never regenerated by the CLI in normal operation. Expose an `IsFirstRun()` helper returning `true` if and only if the file was created during the current invocation. Cover the Windows path via `os.UserCacheDir()`-style resolution.

### Task 3: Persistent telemetry state file

**Verifies:** cli/telemetry#ac:persistent-state-file-shape-rejected

Implement `internal/telemetry/state.go`: read/write `~/.specscore/telemetry.yaml` per REQ:persistent-state-file-shape with the canonical schema-pointer comment on line 1 and three optional keys (`enabled`, `usage-stats`, `crash-reports`). Reject unknown keys at read-time with a stderr warning naming the file + the offending key and disable telemetry for the invocation (do NOT abort the user's command). Reject at write-time with a hard error. Do NOT auto-create the file on first run — absence means "no preference set; use default."

### Task 4: Opt-out signal evaluator

**Verifies:** cli/telemetry#ac:opt-out-precedence

Implement `internal/telemetry/optout.go` as a pure function: input is the resolved signal set (flag value, env vars, CI markers, persistent state from Task 3) plus a `knownChannels []string` parameter passed by the caller; output is per-channel enabled state. The `knownChannels` parameter is taken as a slice argument, NOT imported from the channel-registry package — this keeps Task 4 free of any code-time dependency on Task 5 even though Task 5 produces the slice the caller passes in. Honor the 4-level precedence (flag → env vars → CI auto-disable → persistent state) and the invariant that opt-out always wins — a `--caller` or `SPECSCORE_CALLER` value MUST NOT re-enable a disabled channel.

### Task 5: Channel registry + transmission-timeout wrapper

**Verifies:** cli/telemetry#ac:transmission-hard-timeout-bounded

Implement the `RegisterChannel(name, transmit)` API in `internal/telemetry/`. Guard the registration against any name not in the closed enum (`usage-stats`, `crash-reports` for the MVP); attempting to register an unknown name MUST panic at `init()` (build-time-effectively, not runtime). Wrap every invocation of a registered transmit-fn in a 500 ms hard timeout — on timeout, drop silently, no retry, no caller-visible error. Verify with a unit test that registers two fake channels which block indefinitely on transmit; assert the wrapper returns within 550 ms (500 ms cap + 50 ms slack) and that the user-visible behavior is identical to a fast successful transmit.

### Task 6: Root command wiring — PersistentPreRun and PersistentPostRun

**Verifies:** cli/telemetry#ac:first-run-notice-shown-once, cli/telemetry#ac:first-run-notice-suppressed-in-ci

Wire `internal/cli/root.go`'s PersistentPreRun to: (a) resolve opt-out signals via Task 4, (b) read or create `install_id` via Task 2 — `IsFirstRun()` informs subsequent steps, (c) when `install_id` did not exist AND no CI auto-disable signal is present, print the first-run notice to stderr per REQ:first-run-notice-content (intro line + 2-item channel bullet list + disable instruction line, mentioning the `[channel-id]` placeholder and the `all` sentinel). PersistentPostRun iterates the channel registry and invokes each enabled channel's transmit-fn through the Task 5 wrapper. The install_id MUST still be created when in CI even though the notice is suppressed — so a later interactive run on the same machine does not re-trigger the notice.

### Task 7: `specscore telemetry` subcommand surface

**Verifies:** cli/telemetry#ac:telemetry-subcommand-status, cli/telemetry#ac:telemetry-subcommand-enable-disable

Implement `internal/cli/telemetry.go` with three verbs each taking an optional positional `[channel]` arg: `status`, `enable`, `disable`. `status` consults the channel registry and per-channel enabled state (via Task 4's evaluator), printing one row per channel with the source of the setting; unknown channel name exits `2` with a stderr message listing the known channels. `enable`/`disable` write to `~/.specscore/telemetry.yaml` via Task 3 — no channel arg writes the global `enabled:` key; with a channel arg writes the per-channel override. Per-channel override beats global at evaluation time.

### Task 8: `docs/telemetry.md` skeleton

**Verifies:** cli/telemetry#ac:docs-telemetry-md-skeleton-present

Author `docs/telemetry.md` with the six required `##` headings in order: `Overview`, `Channels`, `Event Schema`, `What We Don't Collect`, `Opt-out`, `Data Retention`. Populate the parent-owned sections: `Overview`, `What We Don't Collect` (enumerate spec content, feature names, file paths, project paths, git remotes, hostnames, usernames, arbitrary user input), and `Opt-out` (env vars, `specscore telemetry disable`, CI auto-disable). Leave the per-channel subsections (`### usage-stats`, `### crash-reports`) as empty placeholders with a one-line stub — the sibling Plans for `usage-telemetry` and `errors-telemetry` populate them. Verify the file lints clean and the AC's required literal strings are present.

## Open Questions

- **Exact prose of the first-run notice (Task 6).** Source Feature REQ:first-run-notice-content fixes the three-line shape, the proper-noun "SpecScore" capitalization, the two channel names, and the literal command/env-var strings. The connective prose gets a copy-review pass during Task 6 implementation. Lean draft is in the source Feature's Outstanding Questions section — implementer should reuse or refine.
- **Test fixture for the boundary-check (Task 1).** The boundary test rejects a "synthetic violation" — exactly how that violation is induced (a hidden-build-tag file? a test that mutates source then restores it? a static AST walk against a fixture-text constant?) is left to the implementer. The simplest form (AST walk over a string constant containing the forbidden import) is preferred.
